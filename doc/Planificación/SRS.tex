\chapter{Introducción.} 

\section{Propósito.}

\indent \indent El objetivo del siguiente documento es la descripción de los requisitos del sistema de SEguimiento de PROyectos SOftware (SEPROSO). Este docuemtno será referencia para el uso posterior por los desarrolladores y en consecuencia por más documentos de planificación, análisis y desarrollo de la apicación.

\section{Ámbito.}

\indent \indent El sistema a desarrollar se denominará SEPROSO. Es una herramienta para la gestión y seguimiento de proyectos software. La interación con la herramienta depende del rol a desempeñar por el usuario de la aplicación, distinguiéndose los siguientes roles:
\begin{itemize}
	\item Administrador.
	\item Jefe de Proyecto.
	\item Desarrollador.
	\item Responsable de personal.\\
\end{itemize}

El sistema atenderá las peticiones de cada uno de ellos en función del trabajo que desempeñana\\.

\chapter{Descripción general.}

\section{Perspectiva del producto.}

\indent \indent La planificación de un proyecto debe afrontarse de manera adecuada para que al final del mismo se pueda hablar de éxito. No se trata de una etapa independiente abordable en un momento concreto del ciclo del proyecto. Es decir, no se puede hablar de un antes y un después al proceso de planificación puesto que según avance el proyecto será necesario modificar tareas, reasignar recursos, etc.\\

Planificar es armonizar dos tipos de elementos muy diferentes entre sí: \textbf{Objetivos} (exigentes, múltiples y difíciles) y \textbf{Recursos} (limitados, costosos y rígidos).\\

Cuando se planifican proyectos software, aparte de una correcta planificación, es muy importante el seguimiento del proyecto debido a los posibles retrasos de algunas fases y/o actividades, y la necesidad de replanificación de las mismas, y la consiguiente replanificación de todo el proyecto. Estas tareas no son sencillas, de ahí la importancia de una herramienta capaz de facilitar todas las tareas descritas.

\section{Funcionalidades del producto.}

\indent \indent SEPROSO permite a un gestor de proyecto el seguimiento temporal de las actividades que se realizan en un proyecto asociado a él, durante el desarrollo del mismo. La herramienta permite al gestor de proyecto la generación de diversidad de informes, a partir de los datos introducidos por los trabajadores. También permitirá obtener informes de los proyectos ya finalizados.\\

En el caso de un proyecto en curso, la herramiente permite la gestión de infomes de seguimiento y uso eficiente del tiempo por los diferentes recursos humanos de los que se dispone. Todo ello se basa en una herramienta web para la facilidad de uso por parte de los trabajadores.\\

\section{Características del usuario.}

\indent \indent Los usuarios de la herramienta deben estar familiarizados con las herramientas web, y principalmente con el proceso de desarrollo de proeyctos software y todos lo que ello engloba, actividades, tareas y artefactos.\\

Además es necesario que el usuario sea un miembro activo del proyecto actual de trabajo, o bien el gestor de proyectos ya finalizados.\\

\section{Restricciones.}

\begin{itemize}
	\item Todos los proyectos deben tener un responsable.
	\item El sistema debe asegurar limitaciones según el usuario y los proyectos que tiene asignados.
	\item Cada desarrollador debe realizar 40 horas semanales.
	\item Una misma persona sólo puede estar ligada cómo máximo a tres proyectos al mismo tiempo.
	\item El número de actividades a desarrollar por una misma persona no excederá de 100 semanales.
\end{itemize}

\section{Dependencias y suposiciones.}

\begin{itemize}
	\item El administrador debe ser capaz de gestionar a todos los usuarios del sistema, pudiendolos dar de alta o de baja, y asignándolos a proyectos.
	\item Existirá un modelo de proceso por defecto únicamente de actividades.
	\item Existirá un calendario por defecto para todos los usuarios creados, siempre y cuando no lo modifiquen éstos.
	\item Se utiilizará una nomeclatura próxima a la de Proceso Unificado.
\end{itemize}

\chapter{Requisitos específicos.}

\section{Requisistos funcionales.}

\indent \indent Debido a la disponibilidad de diferentes roles se tendrán en cuenta una división de requisitos según el rol al cuál afecten.\\

\subsection{Aspectos generales}

\begin{itemize}
	\item (\textbf{G.1}) La unidad de trabajo de la herramienta serán\textbf{ proyectos software}.
	\begin{enumerate}
		\item Se debe almacenar el estado de cada proyecto después de cada modificación de forma estable.
		\item Un proyecto software se divide en \textbf{etapas}.
		\item Cada etapa estará:
		\begin{itemize}
			\item Compuesta por una o varias tareas.
			\item Identificada por nombre, descripción, duración y proyecto asociado.
		\end{itemize}
		\item Una tarea estará:
		\begin{itemize}
			\item Asignada a una o varias personas.
			\item Identificada por nombre, descripción, duración, dependencias y proyecto asociado.
		\end{itemize}
		\item Las actividades se clasificarán según los siguientes modelos:
		\begin{itemize}
			\item Trato con los usuarios (llamadas, citas individuales, etc.).
      \item Reuniones externas.
      \item Reuniones internas.
      \item Lectura de especificaciones y todo tipo de documentación.
      \item Elaboración de documentación (informes, documentos, programas).
      \item Desarrollo de programas.
      \item Revisión de informes, programas, etc.
      \item Verificación de programas.
      \item Formación de usuarios.
      \item Varios (sin clasificar).
		\end{itemize}
	\end{enumerate}
	
	\item (\textbf{G.2}) Cada actividad, asociada a la tarea, pueden estar en uno de los siguiente estados:	
		\begin{itemize}
			\item Pendiente: el jefe de proyecto no ha supervisado el informe que el desarrollador ha emitido sobre dicha actividad. 
			\item Aceptado: el jefe de proyecto ha dado por bueno el informe.
			\item Rechazado: el jefe de proyecto no ha dado por bueno el informe y debe ser corregido.
		\end{itemize}
	\begin{enumerate}
		\item Debe ser posible crear proyectos desde cero o bien definirlos a partir de un modelo de proceso por defecto (plantilla de proyecto).
		\item Un modelo de procedo por defecto será una plantilla en la que unicamente se realiza una división en actividades.
		\item En un proyecto sólo se guardan datos de actividades en curso, que consiste en el seguimiento de tiempo de los recursos asignados a cada una de ellas.
		\begin{itemize}
			\item Sólo se trata con recursos humanos.
			\item Se registran datos de tiempo de lunes a viernes, con un calendario de 40 horas semanales.
		\end{itemize}
		\item No se pueden modificar ni tareas ni etapas de proyectos ya iniciados.
		\item Pueden existir en el sistema tantos proyectos cómo se desee.
		\item Cada proyecto puede seguir un modelo de proceso distinto, definido en su creación.
		\item El número de proyectos simultáneo en los que trabaja un desarrollador es parametrizable.
	\end{enumerate}
	 
	\item (\textbf{G.3}) Existirán \textbf{4 tipos de usuarios}: Administrador, Jefe de proyecto, Desarrollador y Responsable de personal.
	\begin{enumerate}
		\item Sólo existirá un Administrador del sistema y un Responsable de Personal.
		\item Para cada proyecto sólo se le asocia un Jefe de Proyecto.
		\item Los usuarios pueden desepeñar el papel de jefe de proyecto y de desarrollador en un número máximo de proyectos parametrizable.
		\item Cada desarrollador tendrá asignado su rol (o categoría).
		\item Los desarrolladores podrán realizar tareas de categorías iguales o inferiores a la suya.
		\item Las tareas y las etapas pasan por los siguientes estados: no iniciada, activa, finalizada.
		\item Los usuarios podrán editar su perfil:
		\begin{itemize}
			\item Nombre y apellidos.
			\item Contraseña.
			\item Periódo de vacaciones, 4 semanas divididas a los sumo en 3 periódos mínimo de 1 semana.
		\end{itemize}
		\item Cada trabajador desempeñará 40 horas semanales trabajando.
	\end{enumerate}
\end{itemize}


\subsection{Administrador}

\begin{itemize}
	\item (\textbf{A.1}) Registrará las cuentas de los usuarios.
	\item (\textbf{A.2}) Asignará el tipo de cada usuario registrado en el sistema.
	\item (\textbf{A.3}) Cargará los datos iniciales de la herramienta.
	\item (\textbf{A.4}) Cuando se cree un proyecto, asignará el jefe de proyecto.
	\item (\textbf{A.5}) Será el encargado de definir y editar la tabla de roles.
	\item (\textbf{A.6}) Podrá consultar informes destinados al administrador.
	\begin{enumerate}
		\item Registro de acceso al sistema, login y actividad realizada
		\item Estadísticas.
			\begin{itemize}
				\item Número de proyectos registrados, y personas asociadas.
				\item Número de usuarios registrados.
				\item Tabla de roles y usuarios.
			\end{itemize}
	\end{enumerate}
	\item (\textbf{A.7}) Será el encargado de determinar el número de proyectos que es capaz de trabajar un desarrollador simultáneamente.
	
\end{itemize}

\subsection{Jefe de Proyecto}

\begin{itemize}
	\item (\textbf{J.1}) Sólo existe un jefe de proyecto para cada proyecto abierto.
	\item (\textbf{J.2}) Se encarga de seleccionar a las personas asignadas a su proyecto.
	\begin {enumerate}
		\item Sólo puede asignar personal libre de trabajo.
		\item Asigna roles concretos al personal de su proyecto según la tabla de roles de la herramienta.
	\end{enumerate}
	\item (\textbf{J.3}) Si trabaja como desarrollador solo lo podrá hacer en un solo proyecto, distinto al actual.
	\item (\textbf{J.4}) El jefe de proyecto participa en el proyecto que dirige con un tanto por ciento de su tiempo.
	\item (\textbf{J.5}) A la hora de crear el proyecto, asignará los usuarios participantes en el proyecto indicando el porcentaje de participación.
	\begin{enumerate}
		\item El porcentaje de participación de un desarrollador en un proyecto es por defecto y como máximo del 100%.
		\item No podrá asignar un porcentaje de participación que sumado a los del desarrollador en otros proyectos supere el máximo admitido.
	\end{enumerate}
	\item (\textbf{J.6}) Será el encargado de realizar la planificación inicial.
	\item (\textbf{J.7}) Podrá acceder a los siguientes informes:
	\begin{enumerate}
		\item Relación de trabajadores con tareas asignadas en un periodo de tiempo.
		\item Relación de trabajadores y sus tareas asignadas durante un periodo determinado.
		\item Relación de trabajadores con informes de actividad pendientes de supervisión y fechas de los mismos.
		\item Relación de trabajadores con informes de actividad pendientes de aprobación y fechas de los mismos. 
		\item Relación de tareas activas en una fecha o periodo de tiempo, viendo el tiempo planificado y el tiempo realizado de cada una de ellas.
		\item Relación de las tareas finalizadas y activas en una fecha dada (por defecto la actual) que permita apreciar el tiempo planificado y el utilizado realmente en cada una.
		\item Relación de tareas que han consumido o están consumiendo más tiempo del planificado. 
		\item Informe de las tareas a realizar, así como los recursos asignados, durante un periodo de tiempo determinado posterior a la fecha actual.
		\item Relación de personas, con las tareas asignadas y los tiempos de trabajo para cada una de ellas para un periodo de tiempo posterior a la fecha actual (la semana siguiente, la quincena siguiente, etcétera)
		\item En los informes 3 y 4 se podrá acceder a los informes de actividad directamente desde ellos.
	\end{enumerate}
	\item (\textbf{J.8}) Puede acceder a informes de proyectos finalizados.
	\item (\textbf{J.9}) Podrá finalizar etapas activas sin tareas sin finalizar.
	\item (\textbf{J.10}) Al cerrar el proyecto obtendrá informes sobre cómo ha ido la realización del mismo.
\end{itemize}

\subsection{Desarrollador}

\begin{itemize}
	\item (\textbf{D.1}) Introducirá la \textbf{relación de actividades} realizadas en los informes de actividad, para las tareas que tiene asignadas:
	\begin{enumerate}
		\item Los informes de actividad constaran de:
		\begin{itemize}
			\item Tipo de actividad (una o más)
			\item Tiempo dedicado.
		\end{itemize}
		\item Sólo podrá introducir datos de actividades en tareas asignadas que estén activas.
	\end{enumerate}
	\item (\textbf{D.2}) No puede modificar actividades aprobadas por el gestor del proyecto.
	\item (\textbf{D.3}) El tiempo dedicado a actividades en una semana dentro de un proyecto no puede superar su porcentaje de participación en el mismo.
	\item (\textbf{D.4}) Podrá acceder a los siguientes informes.
	\begin{enumerate}
		\item Informe de actividades por semanas
		\item Informe de tiempo dedicado a tareas de proyectos ya finalizados.
	\end{enumerate}
	\item (\textbf{D.5}) El desarrollador podrá trabajar como mucho en 3 proyectos (valor parametrizable), siempre que no sea jefe de proyecto (en tal caso sólo podrá en uno).
\end{itemize}

\subsection{Responsable de personal}
\begin{itemize}
  \item (\textbf{R.1}) Podrá acceder a los siguientes informes:
	\begin{enumerate}
		\item Informe de acumulación del trabajo de cada persona en las diferentes actividades, por semanas o por meses.
	\end{enumerate}
\end{itemize}
